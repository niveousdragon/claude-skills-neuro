# Как клодить: скиллы и рабочие практики для нейробиолога

Мета-гайд для коллег, которые используют **Claude Code** в работе с нейроданными
(calcium imaging, поведение, анализ, фигуры, статьи). Один документ + папка
скиллов. Распаковал, поставил — работаешь.

> Если ты сейчас Claude и читаешь этот файл в чужом проекте: твоя задача —
> помочь коллеге поставить скиллы (`install.ps1` / `install.sh`), а дальше
> вызывать нужный скилл по ситуации и предлагать доустановить плагины из
> раздела [«Что доустановить»](#5-что-доустановить-и-зачем). Подробные триггеры —
> в разделах ниже.

## 1. За 5 минут (TL;DR)

**Шаг 1. Поставить скиллы из этого репозитория.**

Windows (PowerShell):
```powershell
.\install.ps1
```

macOS / Linux:
```bash
./install.sh
```

Скрипт копирует папки из `skills/` в `~/.claude/skills/` (на Windows —
`C:\Users\<имя>\.claude\skills\`). После этого скиллы доступны в любом проекте.

Не хочешь возиться с терминалом? Просто попроси своего Клода:
> «Распакуй скиллы из этого репозитория» — он запустит нужный скрипт сам.

**Шаг 2. Доустановить полезные плагины** (по желанию, но очень рекомендую).
В сессии Claude Code набери:
```text
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers

/plugin marketplace add Imbad0202/academic-research-skills
/plugin install academic-research-skills
```
Подробнее — раздел [5](#5-что-доустановить-и-зачем).

**Шаг 3. Проверить.** Открой Claude Code в любой папке и скажи:
> «Какие скиллы тебе доступны?»

Если в списке есть `deep-analysis` и `figure-alignment` — всё работает.

## 2. Скиллы, команды, агенты, плагины, MCP — что это вообще

Коротко, без жаргона. Это пять разных вещей, которые часто путают.

| Термин | Что это | Аналогия |
|---|---|---|
| **Скилл (Skill)** | Папка с файлом `SKILL.md` — инструкция «как делать X правильно». Клод сам подтягивает её, когда видит подходящую задачу. | Методичка на полке, которую ассистент достаёт, когда нужно. |
| **Команда (slash-команда)** | То, что ты пишешь как `/имя`. Запускает заранее заданный сценарий. | Горячая клавиша. |
| **Агент (subagent)** | Отдельный «Клод», которому делегируют замкнутую подзадачу со своим контекстом. Полезно для параллельной работы и чтобы не засорять основной диалог. | Стажёр, которому дал поручение и ждёшь результат. |
| **Плагин (Plugin)** | Набор скиллов + команд + агентов, который ставится одной командой через marketplace. | Сборник методичек одним пакетом. |
| **MCP** | Протокол, через который Клод ходит во внешние сервисы (bioRxiv, Google Calendar, базы данных). | Розетки, в которые втыкаются внешние инструменты. |

**Чем скилл лучше, чем копипастить инструкцию в чат?**

- **Срабатывает сам.** Не нужно каждый раз вставлять «помни, что фигуры делаем
  на одном GridSpec…». Клод подтянет скилл, когда увидит задачу про фигуры.
- **Не занимает контекст, пока не нужен.** Скилл грузится только в момент
  применения — диалог остаётся чистым.
- **Воспроизводимо и расшаримо.** Один раз написал — пользуется вся лаборатория,
  результат одинаковый у всех.
- **Накапливается.** Нашёл грабли — записал в скилл, больше никто на них не
  наступит.

Грубо: **промпт — это сказать один раз; скилл — это научить навсегда.**

## 3. Как Клод выбирает скилл

У каждого скилла в шапке `SKILL.md` есть поле `description` — короткое описание
**когда** его применять (триггеры, симптомы, ситуации). Клод смотрит на твою
задачу, сопоставляет с описаниями всех доступных скиллов и подтягивает
подходящий. Само тело скилла (как именно делать) грузится только после
срабатывания.

```yaml
---
name: figure-alignment
description: Use when creating, modifying, or debugging multi-panel matplotlib
  figures for publication. Use when panels misalign, axes don't share baselines...
---
```

Практические выводы для тебя:

- **Зови задачу её словами.** «Сделай мультипанельную фигуру для статьи»,
  «панели не выравниваются», «расследуй, почему анализ даёт странный результат» —
  такие формулировки попадают в триггеры и поднимают нужный скилл.
- **Если Клод не подтянул скилл сам** — попроси прямо: «используй скилл
  `figure-alignment`».
- **Описание важнее тела.** Когда пишешь свой скилл (раздел 8), 80% пользы — в
  грамотном `description`: именно по нему скилл находится.

## 4. Наши скиллы

Два скилла лежат в этом репозитории (`skills/`) и ставятся install-скриптом.

### `deep-analysis` — расследование без правок кодовой базы

Изолированная сессия для расследования багов, проблем производительности или
прототипирования алгоритмов. Создаёт отдельную папку `.analysis/`, гоняет
эксперименты, собирает доказательства (метрики, графики) и пишет
**handoff-резюме** с выводами и предложениями — **не трогая основной код**.
Менять код предлагается отдельно, после твоего одобрения.

Когда звать:
- «Почему этот нейрон-фичер MI считается так медленно — профилируй»
- «Сравни KSG и GCMI на синтетике, дай цифры»
- «Разберись, откуда в результате берётся вот эта аномалия»

Пример запроса:
> «Запусти deep-analysis: почему пайплайн INTENSE тормозит на больших
> экспериментах».

### `figure-alignment` — мультипанельные фигуры для публикаций

Сборка фигур из нескольких панелей на **едином master GridSpec**, с
**программной проверкой** выравнивания (а не «на глаз»), квадратности панелей и
перекрытий. Решает классическую боль: панели не на одной базовой линии, «квадрат»
не квадратный, правка одной панели ломает другие.

Когда звать:
- «Собери фигуру 1 для статьи: 5 панелей, A и B в одном ряду»
- «Панели C и D не выровнены по нижнему краю»
- «Сделай панель B физически квадратной»

Пример запроса:
> «Используй figure-alignment: нужна публикационная фигура 18×22 см, слева две
> панели, справа три, всё выровнять и проверить».

## 5. Что доустановить и зачем

Эти наборы мы **не копируем** в репозиторий (у них свои апстримы и лицензии) —
ставь их из официальных marketplace. Раздел заодно служит инструкцией для самого
Клода: если задача коллеги попадает в один из сценариев ниже, предложи поставить
соответствующий плагин.

### `superpowers` — рабочий процесс

16 процессных скиллов: как **думать** над задачей, а не только что печатать.

```text
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers
```

Самые полезные:
- **brainstorming** — прежде чем писать код/фичу, разложить замысел на требования
  и развилки. Зови перед любой новой разработкой.
- **systematic-debugging** — дисциплина отладки: сперва найти корневую причину,
  потом чинить. Зови на любом баге/непонятном поведении.
- **test-driven-development** — тесты до кода.
- **writing-plans** / **executing-plans** — план многошаговой работы и его
  исполнение с чекпойнтами.
- **verification-before-completion** — не объявлять «готово», пока не показал
  доказательства (прогон команды, вывод).

### `academic-research-skills` (ARS) — статьи и литература

Полный пайплайн научной статьи: от плана до проверки цитат. Команды `/ars-*`.

```text
/plugin marketplace add Imbad0202/academic-research-skills
/plugin install academic-research-skills
```

Самые полезные:
- **`/ars-plan`** — сократический разбор структуры статьи по главам.
- **`/ars-lit-review`** — аннотированная библиография по теме.
- **`/ars-citation-check`** — отчёт об ошибках в цитировании (галлюцинации,
  неверные годы/авторы). Критично: модели выдумывают ссылки.
- **`/ars-reviewer`** — симуляция peer-review панели.
- **`/ars-format-convert`** — конвертация между LaTeX / DOCX / PDF / Markdown.

> Лицензия ARS — CC BY-NC 4.0 (некоммерческое использование). Для академической
> работы это ок.

### `deep-research` — многоисточниковое исследование

Если установлен — харнесс для глубокого ресёрча: веер веб-поисков, чтение
источников, проверка фактов, сборка отчёта со ссылками. Зови, когда нужен
основательный обзор по теме с проверяемыми источниками, а не быстрый ответ.

## 6. Рецепты нейробиолога

Готовые сценарии. Скопируй формулировку, подставь своё.

**Фигура для статьи**
> «Собери публикационную фигуру: <сколько> панелей, <раскладка>. Используй
> `figure-alignment`, проверь выравнивание программно, не на глаз».

**Расследовать странный результат анализа**
> «Запусти `deep-analysis`: <что не так>. Нужны цифры и графики, код не трогай —
> только предложи изменения».

**Профилировать медленный пайплайн**
> «`deep-analysis`: профилируй <функцию/пайплайн>, найди узкие места, сравни
> альтернативы по времени и качеству».

**Черновик секции по литературе**
> «`/ars-lit-review` по теме <…>» — затем — «`/ars-plan`, помоги выстроить
> структуру раздела Methods».

**Проверить цитаты перед сабмишном**
> «`/ars-citation-check`: проверь все ссылки в <файл .bib / .tex> на выдуманные,
> неверные годы и авторов».

**Разобрать чужой код пайплайна**
> «Объясни, что делает <модуль>, нарисуй поток данных, отметь подозрительные
> места. Не меняй код».

**Спланировать новую фичу анализа**
> «Используй `brainstorming`: хочу добавить <фича>. Разложи на требования и
> развилки до кода».

## 7. Принципы «как клодить»

Семь привычек, которые сильнее всего поднимают качество работы с Клодом.

1. **Заведи `CLAUDE.md` в проекте.** Это память проекта: как запускать тесты,
   соглашения, что нельзя трогать, структура данных. Клод читает её каждую
   сессию. Один раз вложился — экономишь часы объяснений.

2. **Давай контекст, а не только команду.** «Почини это» — слабо. «Вот ошибка,
   вот что я ждал, вот как воспроизвести» — сильно. Контекст экономит итерации.

3. **Требуй доказательства, а не обещания.** «Готово» без прогона теста ничего не
   значит. Проси показать вывод команды, цифры, до/после. (Плагин `superpowers`
   зашивает это в `verification-before-completion`.)

4. **Не давай выдумывать числа и ссылки.** Модели галлюцинируют статистику и
   цитаты. Правило: любая цифра — из реального прогона; любая ссылка — проверена
   (`/ars-citation-check`). Если данных нет — пусть скажет «не знаю», а не
   придумает.

5. **Итерируй маленькими шагами.** Большой запрос «сделай всё» → много скрытых
   допущений. Лучше: разбить, согласовать план (`writing-plans`), идти шагами с
   проверкой.

6. **Безопасные правки.** Для тяжёлых расследований используй `deep-analysis` —
   он не трогает рабочий код. Перед массовыми изменениями проси сперва план, а
   правки — в отдельной ветке.

7. **Записывай грабли в скиллы и `CLAUDE.md`.** Наступил на проблему один раз —
   зафиксируй, чтобы лаборатория не наступала снова. Так знание накапливается, а
   не теряется в истории чатов.

## 8. Хочу свой скилл

Скилл — это переиспользуемая методичка по приёму, который ты не хочешь объяснять
каждый раз заново. Создаётся за минуты.

**Когда стоит писать скилл:**
- приём был неочевиден, ты бы вернулся к нему ещё не раз;
- применимо широко (не к одному проекту — тогда это `CLAUDE.md`);
- польза для коллег.

**Не стоит:** разовые решения; то, что и так хорошо документировано; чисто
проектные соглашения (им место в `CLAUDE.md`).

**Анатомия скилла** — одна папка, один файл:
```
~/.claude/skills/
  мой-скилл/
    SKILL.md          # обязательный
    helper.py         # опционально: скрипты/шаблоны, если нужны
```

**Минимальный `SKILL.md`:**
```markdown
---
name: имя-через-дефис
description: Use when <конкретные триггеры и симптомы, когда применять>.
---

# Название скилла

## Overview
Что это и главный принцип в 1–2 предложениях.

## When to Use
- симптом / ситуация 1
- симптом / ситуация 2

## Workflow
Пошагово, как делать правильно. Примеры кода — инлайн, если короткие.
```

**Главное правило — `description`.** Именно по нему скилл находится (раздел 3).
Пиши от третьего лица, начинай с «Use when…», перечисляй **конкретные триггеры и
симптомы**, а не пересказ процесса. Хорошо: «Use when panels misalign, axes don't
share baselines, squares aren't square». Плохо: «Helps with figures».

**Как поделиться с лабой:** положи папку скилла в `skills/` этого репозитория,
закоммить — у всех появится после `git pull` + повторного `install`.

**Хочешь по-серьёзному** (с тестированием скилла на субагентах, по TDD):
- поставь `superpowers` и вызови скилл **`writing-skills`** — это официальный
  процесс создания и проверки скиллов;
- либо плагин **`skill-creator`** (`/plugin install skill-creator`) — мастер,
  который проведёт за руку.
